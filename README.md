## Ответы на вопросы (Interview Corner)
1. Почему в JSX используется className, а не class?
JSX — это не HTML, а замаскированный JavaScript. В JavaScript слово class зарезервировано для создания классов (ООП). Чтобы движок не путался, создатели React решили использовать className для назначения CSS-классов.

2. Чем props отличаются от state?
props (пропсы) — это внешние настройки, которые приходят в компонент от родителя. Они предназначены только для чтения. state (состояние) — это внутренняя память самого компонента. Компонент может сам изменять свой стейт в ответ на действия пользователя.

3. Почему props нельзя изменять внутри компонента?
React построен на принципе однонаправленного потока данных (сверху вниз). Пропсы принадлежат родителю. Если компонент начнет втихаря менять пропсы, возникнет хаос, и отследить ошибки будет невозможно. Если нужны изменения — используем state.

4. Почему state нужно изменять через setter-функцию, а не прямым присваиванием?
Если написать note = "текст", переменная изменится, но React об этом не узнает и не обновит экран. Setter-функция (например, setNote) меняет значение и дает команду React'у: "Данные изменились, срочно перерисуй этот компонент!".

5. Что такое управляемый компонент и почему он важен для форм?
Это инпут, значение которого (value) жестко привязано к React-стейту, а изменения происходят через onChange. Это нужно, чтобы React всегда был "единым источником истины". Что в стейте — то и на экране. Это защищает от рассинхронизации данных и интерфейса.

6. Как Вы понимаете формулу UI = f(state)?
Пользовательский интерфейс (UI) — это результат функции от текущего состояния (state). То есть экран — это просто визуальное отражение данных. Мы просто меняем стейт, а интерфейс сам автоматически перестраивается под новые данные.

7. Зачем компонентам в React нужен key при рендеринге списка через map()?
key — это уникальный идентификатор (как номер паспорта). Когда элементов в списке много, key помогает React'у понять, какая именно карточка изменилась, добавилась или удалилась, чтобы перерисовать только её, а не тратить ресурсы на перерисовку всего списка заново.

## Этап 5: Связь с будущим курсовым проектом
Данная практика является прямым прототипом для моего будущего SPA «Интерактивная база образовательного контента».
В ходе работы была заложена базовая архитектура:

Декомпозиция: создан переиспользуемый UI-компонент ResourceCard для отображения учебных материалов и логический контейнер ResourceList для рендеринга списка.

Props: настроена передача данных (уровни английского языка, форматы уроков) из главного родительского файла в карточки.

State: добавлено локальное состояние (useState) для симуляции добавления материала в «Подборки к уроку» и добавления личных заметок преподавателя, что напрямую отражает бизнес-логику моей курсовой работы.
